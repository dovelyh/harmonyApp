# HarmonyOS NEXT 迁移、生态与上架踩坑

## 1. 与 Android 的根本差异（社区共识）

- **HarmonyOS NEXT** 不再兼容 **Android APK** 直接安装运行；应用需以 **ArkTS + ArkUI** 等原生范式 **重写或大规模迁移**。
- **JVM / ART** 不再作为应用运行时；Java/Kotlin 业务逻辑需 **迁到 ArkTS** 或 **C/C++ NDK** 等方案，并重新对接系统 API。
- **权限**：由 `AndroidManifest.xml` 迁移到 **`module.json5`** + **运行时授权**（`abilityAccessCtrl` 等模块），模型与安卓不完全相同。

*来源：CSDN/IT 之家《HarmonyOS NEXT 与安卓开发对比及迁移指南》、开发者社区《鸿蒙 Next 适配避坑全攻略》等。*

## 2. 迁移中的高频陷阱（社区文章归纳）

| 领域 | 常见坑 | 方向性对策 |
|------|--------|------------|
| 线程与后台 | 沿用安卓「常驻后台」思路易碰壁 | 使用系统允许的 **任务调度、推送、短任务**；长连接需评估功耗与规范 |
| 存储路径 | 沙箱路径与安卓不同 | 使用鸿蒙文档中的 **应用文件目录**、**preferences** 等 API |
| 第三方库 | 依赖 Android 专有 API 的库无法直接用 | 在 **ohpm** 找鸿蒙适配版或自研替换；**`.so` 架构** 需 **arm64-v8a** 等与目标机一致 |
| UI | XML → 声明式 ArkUI | 布局与状态需 **重写**，非机械翻译 |
| 广播等 | 安卓 Broadcast 等无直接等价 | 按鸿蒙 **Ability/事件** 模型重做 |

## 3. ABI 与原生库

- 真机多为 **arm64-v8a**；若三方 **`.so` 仅 armeabi-v7a**，可能安装或运行失败；需在 **`build-profile`** 等配置中 **声明支持的 ABI** 或更换库。

*来源：《100 个项目疑难杂症汇编》打包篇等。*

## 4. 应用上架与审核

- **隐私合规**：在用户同意隐私政策 **之前** 初始化 SDK、读取 **MAC/设备标识** 等，易导致 **审核驳回**；需延迟初始化到同意后。
- **HAP 体积**：单包超限需 **HSP 分包**、资源优化等。
- **Profile / APP ID**：**bundleName**、证书与开发者后台配置必须 **一致**，否则安装或发布校验失败。
- **混淆**：若使用 **反射或动态属性名**，混淆后可能 **运行时崩溃**；需 **keep** 规则（如保留属性名，具体以工具文档为准）。

## 5. 用户数据与系统升级（资讯向）

- 多篇资讯提到升级 NEXT 时应用数据可通过 **云空间** 等方案迁移，属 **用户侧** 说明；开发者仍需在应用内做好 **备份与恢复** 设计。

## 6. 生态与文档（社区情绪向，仅供参考）

部分开发者反馈：**API/工具迭代快**、**文档与 Demo 滞后**、**示例与真实行为不一致** 会拉长学习曲线；对策是 **锁定 SDK 版本**、以 **官方文档 + issue/论坛** 交叉验证、关键路径写 **最小复现** 再求助。
