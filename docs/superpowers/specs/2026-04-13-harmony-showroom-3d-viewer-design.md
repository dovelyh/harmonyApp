# HarmonyOS 展馆页 3D 车模查看器设计说明

## 项目目标

在当前 HarmonyOS ArkTS 应用中新增一个底部栏目 `展馆`，用于承载类似懂车帝车型展示的 3D 查看体验。这个页面需要同时满足三类目标：

1. 用户可以在底部导航中直接进入 `展馆` 页面。
2. 页面中心区域可以展示一个可自动旋转、可拖拽查看角度、可缩放的 3D 车模。
3. 当前实现应优先兼顾效果、开发速度和后续扩展性，保证后面可以替换为更高精度的正式模型资源。

## 设计结论

本次采用 `ArkUI 页面壳 + Web 组件承载本地 3D Viewer` 的方案，而不是直接在 ArkUI 中伪造 3D 效果，也不在第一阶段就走 `XComponent + OpenGL` 原生渲染。

选择这个方案的原因有三点：

1. 它能提供真正的 3D 透视、旋转和缩放体验，比 ArkUI 的旋转动画更接近懂车帝那类车型展示。
2. 它保留了 ArkUI 作为主页面框架，新增成本较低，能够快速接入现有 Tab 结构和视觉体系。
3. 它天然适合后续替换模型，只需要替换 `glb/gltf` 资源和少量配置，不需要重写整个页面架构。

## 方案对比

### 方案一：ArkUI 伪 3D 动画

优点是接入最快、兼容性高；缺点是只能做“像 3D”的视觉效果，无法达到真实模型查看器的体验。本方案不满足当前“像懂车帝一样展示车模”的目标。

### 方案二：ArkUI + Web 本地 3D Viewer

优点是效果真实、开发效率高、后续替换模型成本低。本次选择此方案。

### 方案三：XComponent + OpenGL 原生渲染

优点是控制力最强，适合大型 3D 项目；缺点是开发量、调试复杂度和维护成本都明显更高。对当前这个以内容展示为主的 App 来说，这一阶段投入过重，因此暂不作为首版实现。

## 信息架构

底部导航从当前 4 个 Tab 扩展为 5 个 Tab，推荐顺序如下：

1. 首页
2. 发现
3. 展馆
4. 消息
5. 我的

把 `展馆` 放在中间位置，有两个考虑：

1. 3D 车模展示是这次新增的核心亮点，放在中间更容易被发现。
2. 中间位置更像一个“内容中心”入口，符合展馆的定位。

## 页面结构

`展馆` 页面采用单页纵向结构，从上到下分成四个区域：

### 1. 顶部标题区

顶部展示 `展馆` 标题和一句副标题，例如“360° 浏览数字车模，感受细节与比例”。这一层只负责说明页面用途，不放复杂操作，保证首次进入时的理解成本足够低。

### 2. 3D Viewer 主展示区

页面核心区域是一张大卡片，内部放置 Web 组件承载的 3D Viewer。主展示区具备以下行为：

1. 页面进入后车模自动慢速旋转，帮助用户感知立体结构。
2. 用户可以手势拖拽改变观察角度。
3. 用户可以缩放查看局部细节。
4. Viewer 加载期间展示原生加载态。
5. Viewer 加载失败时展示原生降级卡片和重试提示。

这一层是整页的视觉重心，卡片本身仍保留当前应用的圆角、浅底和留白风格，避免 3D 区域和现有页面显得割裂。

### 3. 展品信息区

3D 主展示区下方放一张信息卡，内容包括：

1. 车型名称
2. 展品标签，例如“数字孪生”“可交互模型”“本地资源”
3. 一段简短说明，介绍该模型当前用于展示交互能力和页面结构

这样页面不会只有一个孤立模型，而是形成“展品 + 说明”的完整内容体验。

### 4. 操作提示区

页面底部放一行轻量提示，例如“拖动可旋转，双指可缩放”。这样能降低首次使用门槛，也方便后续扩展更多手势说明。

## 技术架构

本次实现分为四个层次：

### 1. ArkUI 容器层

新增 `ShowroomPage.ets` 作为 `展馆` 页面入口，并在 `Index.ets` 中作为新的 `TabContent` 接入到底部导航。页面负责标题、卡片容器、原生加载态、错误态和展品说明区。

### 2. Web Viewer 承载层

在 ArkUI 页面中使用 Web 组件加载本地 HTML Viewer。Viewer 所需的 HTML、CSS、JavaScript 资源统一放在 `entry/src/main/resources/rawfile/` 下，便于应用离线打包和本地读取。

### 3. 3D 渲染层

本地 HTML 内使用 WebGL 方案渲染 3D 车模。首版优先选择成熟的前端 3D 运行方式来实现：

1. 初始化场景、相机、灯光和地面阴影
2. 加载本地车模资源或首版演示模型
3. 提供自动旋转和手势轨道控制
4. 在模型加载失败时回退到明确的错误提示

### 4. ArkUI 与 Viewer 状态桥接层

ArkUI 页面需要知道 Viewer 是否已准备完成，因此要建立一个轻量状态桥接：

1. Viewer 初始化完成后，通知 ArkUI 隐藏加载态。
2. Viewer 资源加载失败时，通知 ArkUI 展示错误态。
3. 后续如果需要切换车型，也可以沿用这一层传递模型配置。

## 文件设计

本次设计建议新增或调整的核心文件如下：

1. `entry/src/main/ets/pages/Index.ets`
   增加 `展馆` Tab 和对应图标、顺序、激活态逻辑。
2. `entry/src/main/ets/pages/ShowroomPage.ets`
   新增展馆主页面。
3. `entry/src/main/resources/base/profile/main_pages.json`
   注册新增页面路径。
4. `entry/src/main/resources/rawfile/showroom/index.html`
   本地 3D Viewer 入口页面。
5. `entry/src/main/resources/rawfile/showroom/`
   存放 Viewer 的脚本、样式和模型资源。

如果实现时页面结构需要更清晰，还可以补充一个页面专属子组件目录，例如 `entry/src/main/ets/views/showroom/`，把信息卡、加载态等拆成小组件，但这一点不是首版硬要求。

## 数据与状态流

当前 `展馆` 页不依赖后端接口，采用本地静态配置即可。状态流分为两段：

### ArkUI 页面内状态

页面至少维护下列状态：

1. 当前 Viewer 是否正在加载
2. 当前 Viewer 是否加载失败
3. 当前展品的标题、描述和标签

这些状态由 `ShowroomPage.ets` 自己维护，不需要接入现有 API 层。

### Web Viewer 内状态

Viewer 内部维护：

1. 场景初始化状态
2. 模型加载状态
3. 相机控制和自动旋转状态

Viewer 内部只负责渲染，不负责页面文案和布局。这样可以把“页面壳”和“模型展示器”清晰分开，后续替换模型或调整视觉时影响面更小。

## 模型资源策略

当前用户希望效果尽量接近懂车帝式车型展示，因此模型资源策略采用两阶段：

1. 首版先接入一个可用的本地车模演示资源，目标是先跑通真实 3D 交互链路。
2. 后续如果用户提供正式车型的 `glb/gltf` 模型，只替换资源文件和少量配置，不改页面结构。

首版不把“接入复杂模型管理能力”作为范围要求，避免为了扩展性过度设计。

## 错误处理与降级

页面必须考虑 Web 组件和 3D 资源的异常情况，避免出现空白屏：

1. 如果 Viewer 尚未准备完成，展示 ArkUI 原生加载态。
2. 如果本地模型资源加载失败，展示“展品加载失败，请稍后重试”的错误卡片。
3. 如果 Web 组件无法正常加载，页面仍应保留标题区和信息区，不影响整个 App 的 Tab 切换。

这里要特别注意一个平台约束：HarmonyOS 官方 WebView Codelab 明确提到 `Previewer 不支持 WebView 效果预览`，因此 3D Viewer 的最终效果验证应在模拟器或真机完成，而不能只依赖预览器结果。这是本次实现必须接受的调试现实。

## 视觉原则

`展馆` 页面虽然承载 3D 能力，但仍应继承当前应用的整体风格：

1. 页面背景继续使用浅灰底，和首页、发现页一致。
2. 主要内容使用圆角白卡片承载，保持现有卡片体系。
3. 3D 展示区可以更深色、更聚焦，用来提升科技感，但外围布局仍保持整洁和留白。
4. 文案和信息卡尽量简洁，不抢 3D 模型本身的视觉主角地位。

## 测试与验收

本次实现完成后，至少要验证下面几类内容：

1. `Index` 页面底部可以看到并切换到 `展馆` Tab。
2. `展馆` 页面可以稳定展示标题区、主展示区和信息区。
3. 3D 模型在模拟器或真机上可以正常加载。
4. 模型支持自动旋转。
5. 用户拖拽和缩放时，模型视角能够变化。
6. 模型加载失败时，页面能显示原生错误态而不是白屏。
7. 项目构建命令可以正常通过。

由于现有项目测试基础主要是 Hypium 示例，首版自动化测试应优先覆盖可抽离的轻逻辑，例如展馆标签数据、Tab 配置或状态辅助函数；而 3D Viewer 的核心验收仍以构建验证和设备实测为主。

## 本轮实现范围

本轮只做“首个可用版本”，范围限定如下：

1. 增加底部 `展馆` Tab。
2. 新增 `展馆` 页面。
3. 接入本地 3D Viewer。
4. 跑通一个可旋转、可拖拽、可缩放的 3D 车模展示。
5. 增加基础加载态、错误态和展品说明。

以下内容明确不放入本轮范围：

1. 多车型切换
2. 联网拉取远程模型
3. 高级材质配置后台化
4. 原生 OpenGL / XComponent 重构

## 验收标准

只要同时满足下面条件，就视为本轮设计达成目标：

1. 用户可以从底部导航进入 `展馆`。
2. 页面核心区域呈现真实 3D 物体，而不是伪 3D 动画。
3. 交互方式至少包含自动旋转、拖拽查看和缩放查看。
4. 页面风格与现有 ArkUI 应用一致，不显得像嵌入了一个割裂的独立网页。
5. 后续替换正式 `glb/gltf` 模型时，不需要改动页面整体架构。
