# HarmonyOS ArkTS 学习中心设计说明

## 项目目标

把当前以首页单页讲解为主的 HarmonyOS 示例工程，升级成一个可以长期扩展的 ArkTS 学习型项目。项目完成后，应当同时满足下面三类学习诉求：

1. 初学者可以通过首页学习地图快速理解 HarmonyOS 日常开发要学什么。
2. 进阶学习者可以进入专题页，按主题系统学习 ArkTS 语法、状态管理、页面导航、数据流和高频 UI 组件。
3. 实战学习者可以进入案例市场，通过资讯、商城、设置、数据看板等页面看到常见业务场景是怎样组织页面、状态和交互的。

## 设计原则

### 1. 保留现有成果

当前 `entry/src/main/ets/pages/Index.ets` 已经包含了大量高质量注释和基础组件示例。新的设计不推翻这份内容，而是把它升级成“首页 + 基础实验区”的双重角色：

1. 顶部增加学习总览、模块导航、阶段说明和案例入口。
2. 中下部保留现有 Tabs 式组件讲解区，作为基础组件实验室。

### 2. 从单页演示升级为多页学习结构

项目新增多个专题页，每个页面关注一个清晰主题，避免单文件无限膨胀：

1. ArkTS 语法页
2. 状态管理页
3. 组件补充页
4. 页面导航与生命周期页
5. 数据与业务页
6. 案例市场页
7. 资讯案例页
8. 商城案例页
9. 设置案例页
10. 数据看板案例页

### 3. 示例必须“可运行、可注释、可迁移”

每个专题页和案例页都必须具备：

1. 模拟数据，可直接预览，不依赖后端。
2. 详细中文注释，解释该页面中 ArkTS / ArkUI 关键语法和常见用法。
3. 可迁移的页面结构，让学习者能把示例拆出来复用到自己的项目。

## 信息架构

### 首页

首页承担五个职责：

1. 展示项目定位和学习目标。
2. 展示学习路径和模块分类。
3. 提供专题页入口。
4. 提供案例市场入口。
5. 保留现有组件实验区 Tabs。

### 专题页

专题页统一采用以下结构：

1. 顶部返回头部，说明当前主题学习目标。
2. 多个白色卡片分节讲解，每节都包含说明、交互示例和重点提示。
3. 页面内部使用模拟数据和状态演示真实开发写法。

### 案例页

案例页统一强调业务感和页面组织方式：

1. 资讯案例：分类筛选、文章列表、标签和推荐位。
2. 商城案例：商品卡片、分类过滤、购物车数量变化、订单摘要。
3. 设置案例：设置分组、开关项、信息项、风险操作区。
4. 数据看板案例：指标卡、任务进度、趋势条、排行榜。

## 技术覆盖范围

本次扩展至少覆盖下列日常高频能力：

1. ArkTS 基础语法：类型、接口、枚举、函数、数组对象操作、模板字符串、条件分支、循环、异步、错误处理。
2. ArkUI 状态能力：`@State`、`@Prop`、`@Link`、`@Provide`、`@Consume`、`@Builder`、`@Styles`、`@Extend`。
3. 页面组织：页面路由、参数传递、返回、生命周期。
4. 高频 UI 组件：Text、Button、TextInput、Toggle、Checkbox、Radio、Slider、Progress、Rating、Badge、Image、List、Grid、Tabs、Scroll、Swiper、LoadingProgress。
5. 列表与业务数据：筛选、排序、加载、空状态、错误状态、统计汇总。
6. 动画与交互：点击、触摸、过渡、属性动画、反馈文案更新。

## 风险与约束

### 1. 编译稳定性优先

项目必须能够在 HarmonyOS 预览器正常编译。因此优先采用当前工程和 API 22 下最稳定、最常见的写法，避免为了“展示更多 API”而引入高风险能力。

### 2. 注释密度较高

用户明确要求“所有代码使用加注释详细说明”。因此页面代码会比纯业务项目更长，但这是刻意保留的教学特征，而不是冗余。

### 3. 先完整骨架，再持续补深度

本次先搭建完整学习骨架和首批核心内容，后续可以继续追加新的专题和能力页，而不需要再次推翻架构。

## 本轮实现范围

本轮优先完成以下内容：

1. 首页升级为学习总控台。
2. 新增公共学习数据与筛选辅助函数。
3. 新增多个专题页与案例页。
4. 更新页面路由配置。
5. 增加根目录变更说明文档。
6. 跑通单元测试与构建命令，确认可编译。

## 验收标准

1. 首页可以进入多个专题页和案例页。
2. 每个新增页面都能独立预览并包含中文详细注释。
3. 至少形成“基础组件 + 专题知识 + 场景案例”三层学习结构。
4. 项目可以通过本地构建验证。
5. 根目录有一份文件级变更说明文档，便于用户审阅新增、修改、删除内容。
